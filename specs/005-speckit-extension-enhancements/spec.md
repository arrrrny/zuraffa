# Feature Specification: Speckit Extension Enhancement - Init, Generate, and Manifest

**Feature Branch**: `004-speckit-extension-enhancements`  
**Created**: 2026-04-26  
**Status**: Draft  
**Input**: User description: "Zuraffa Speckit enhancements: 1) Create speckit.zuraffa.init command for pre-hook setup, 2) Create speckit.zuraffa.generate-commands command for manifest regeneration, 3) Create extension.yml manifest, 4) Move extension to .specify/extensions/zuraffa/"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - One-Click Project Setup (Priority: P1)

An AI agent starts working on a new Flutter project with Zuraffa. Instead of manually editing pubspec.yaml, adding dependencies, creating build.yaml, and adding dependency overrides, the agent runs a single command that sets up everything.

**Acceptance Scenarios**:
1. **Given** a fresh Flutter project, **When** the agent runs `speckit.zuraffa.init`, **Then** pubspec.yaml is updated with all required dependencies, overrides, and build.yaml is created.
2. **Given** an existing project with partial setup, **When** the agent runs the init command, **Then** missing pieces are added without duplicating existing entries.

### User Story 2 - Auto-Regenerate Commands from Manifest (Priority: P1)

When zfa CLI adds new plugin commands or changes flag schemas, the extension commands can be regenerated by running a single command that parses `zfa manifest` JSON.

**Acceptance Scenarios**:
1. **Given** the `zfa manifest` command outputs current plugin schemas, **When** the agent runs `speckit.zuraffa.generate-commands`, **Then** all command .md files in the extension are updated to match.
2. **Given** a new plugin is added to zfa, **When** the generate command runs, **Then** a new .md file appears for the new plugin command.

### User Story 3 - Extension Manifest with Aliases (Priority: P2)

The extension has a machine-readable `extension.yml` that maps all commands with human-readable aliases, enabling IDE integration and command discovery.

**Acceptance Scenarios**:
1. **Given** the extension.yml exists, **When** an IDE loads it, **Then** all 40+ commands are listed with descriptions and aliases.
2. **Given** a developer types a shorthand alias, **Then** the extension maps it to the full command.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `speckit.zuraffa.init` MUST add `zuraffa`, `zorphy_annotation` to dependencies and `build_runner` to dev_dependencies
- **FR-002**: `speckit.zuraffa.init` MUST add `meta`, `analyzer`, `dart_style` dependency overrides for Flutter SDK compatibility
- **FR-003**: `speckit.zuraffa.init` MUST create `build.yaml` with zorphy and json_serializable builder configuration
- **FR-004**: `speckit.zuraffa.init` MUST be idempotent - running twice should not duplicate entries
- **FR-005**: `speckit.zuraffa.generate-commands` MUST parse `zfa manifest` JSON output
- **FR-006**: `speckit.zuraffa.generate-commands` MUST create one .md file per plugin command with frontmatter, usage, flags table, and examples
- **FR-007**: `speckit.zuraffa.generate-commands` MUST organize files into category subdirectories (data, domain, presentation, scaffolding, testing, utilities)
- **FR-008**: `extension.yml` MUST list all commands with name, file path, description, category, and aliases
- **FR-009**: The extension MUST live at `.specify/extensions/zuraffa/` with commands in `commands/` subdirectory
- **FR-010**: All commands MUST support `--dry-run` to preview changes without writing files

### Key Entities

- **ExtensionCommand**: A markdown file with YAML frontmatter containing command metadata
- **CommandRegistry**: JSON mapping of plugin commands to extension files
- **ExtensionManifest**: extension.yml with all command definitions and aliases

## Success Criteria *(mandatory)*

- **SC-001**: Running `speckit.zuraffa.init` on a fresh Flutter project produces a working zfa environment in under 30 seconds
- **SC-002**: `speckit.zuraffa.generate-commands` produces exactly one .md file per plugin command from `zfa manifest`
- **SC-003**: `extension.yml` contains all commands with valid file paths and unique aliases
- **SC-004**: Extension files are relocatable to `.specify/extensions/zuraffa/` without path changes
- **SC-005**: All generated command files follow the same format with consistent frontmatter structure

## Assumptions

- `zfa` CLI is globally installed and `zfa manifest` produces valid JSON
- The extension framework reads .md files from `commands/` subdirectories
- YAML frontmatter is the standard metadata format for speckit extension commands
- Category names are stable and derived from plugin names
