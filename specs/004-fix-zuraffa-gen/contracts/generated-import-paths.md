# Contract: Generated Import Path Format

**Feature Branch**: `004-fix-zuraffa-gen`
**Date**: 2026-04-17

## Overview

This contract defines the format of import paths in all Dart files generated by the Zuraffa CLI (`zfa generate`) and MCP server. It serves as a guarantee to consumers of generated code about what import statements will look like.

## Import Path Rules

### Rule 1: Relative Paths for Project-Local Files

All imports that reference files within the target project MUST use relative paths computed from the generated file's location.

**Format**: `{relative_path}/{target_file}.dart`

Where `{relative_path}` is computed using standard path resolution:
- From `domain/usecases/{domain}/` to `domain/entities/{entity}/` → `../../entities/{entity}/`
- From `data/datasources/{entity}/` to `domain/entities/{entity}/` → `../../../domain/entities/{entity}/`
- From `presentation/pages/{domain}/` to `domain/entities/{entity}/` → `../../../domain/entities/{entity}/`
- From `data/di/{subdir}/` to `domain/{...}/` → `../../domain/{...}/`

**Examples**:

```dart
// Use case file at lib/src/domain/usecases/product/get_product_usecase.dart
import '../../entities/product/product.dart';
import '../../repositories/product_repository.dart';

// Presenter file at lib/src/presentation/pages/product/product_presenter.dart
import '../../../domain/entities/product/product.dart';
import '../../../domain/usecases/product/get_product_usecase.dart';

// DI file at lib/src/data/di/usecases/get_product_usecase_di.dart
import '../../domain/usecases/product/get_product_usecase.dart';
import '../../domain/repositories/product_repository.dart';
```

### Rule 2: Package Imports for External Dependencies Only

Package-style imports (`package:name/...`) are ONLY used for external packages, never for project-local files.

**Allowed package imports**:
- `package:zuraffa/zuraffa.dart` — the Zuraffa framework itself
- `package:flutter/material.dart` — Flutter framework (in views)
- `package:hive_ce_flutter/hive_ce_flutter.dart` — Hive (in local datasources with caching)
- Other external dependencies as needed

**Forbidden package imports**:
- `package:{project_name}/...` — NEVER for project-local files, regardless of whether the package name is known
- `package:app/...` — NEVER (this was the broken fallback)

### Rule 3: Same-Directory Imports

Files in the same directory use bare relative imports without path traversal:

```dart
// In product_presenter.dart
import 'product_controller.dart';
import 'product_state.dart';
```

## Affected Generators

| Generator | Output Depth | Previous Behavior | New Behavior |
|-----------|-------------|-------------------|-------------|
| `CommonPatterns.entityImports()` | varies | `package:{name}/domain/entities/...` | Relative path computed from output file |
| `entity_usecase_generator` | 2 | Mixed (package + relative) | All relative |
| `stream_usecase_generator` | 2 | Mixed (package + relative) | All relative |
| `presenter_plugin` | 3 | Mixed (package + relative) | All relative |
| `state_builder` | 3 | Mixed (package + relative) | All relative |
| `controller_plugin_utils` | 3 | Package imports via CommonPatterns | All relative |
| `interface_generator` (repository) | 2 | Package imports via CommonPatterns | All relative |
| `implementation_generator_append` | 2 | Package imports via CommonPatterns | All relative |

## Generators Already Compliant (No Changes Needed)

- `remote_generator.dart` — already uses relative paths
- `local_generator.dart` — already uses relative paths
- `view_plugin.dart` — already uses relative paths
- `mock_datasource_builder.dart` — already uses relative paths
- `mock_provider_builder.dart` — already uses relative paths
- `di_plugin.dart` — already uses relative paths
- `test_builder_entity.dart` — already uses `path.relative()` dynamically

## Breaking Changes

**None for end users.** The generated code's public API (class names, method signatures) does not change. Only the import statements in generated files change from `package:` to relative paths, which is transparent to consumers of the generated code.

**For the Zuraffa codebase itself**: `PackageUtils.getBaseImport()` and `getPackageName()` may be deprecated or removed if no longer needed after migration.
