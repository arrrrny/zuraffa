# Zuraffa v5 Concrete Action Plan

This plan turns the v5 roadmap into concrete work.

It is intentionally organized into:
- **MUST** — required to call v5 complete
- **SHOULD** — high-value follow-ups for v5.x
- **NICE TO HAVE** — useful but not blocking

---

## Definition of done for v5

V5 is complete only when:

1. all AI-facing surfaces teach the same pipeline
2. active/public repo surfaces no longer teach legacy generation workflows
3. `.zfa/` is intentionally useful, not just present
4. MCP drives the same canonical flow as the CLI
5. `zik_zak` has a documented and executable migration path
6. at least one downstream feature slice is proven through the pipeline

---

# MUST

## M1. Release integrity

### Goal
Align the release identity with the actual shipped v5 product.

### Tasks
- bump package version and release messaging to the real v5 boundary
- align README, pubspec, website, migration docs, and changelog wording
- clearly document what was removed, what is canonical, and what is transitional

### Exit criteria
- no public contradiction between package version and v5 docs
- release notes and migration notes describe the same contract

---

## M2. Canonical AI pipeline everywhere

### Goal
Every AI agent should be forced into the same generation contract.

### Canonical contract
1. `zfa entity create`
2. `zfa make`
3. `zfa build`

### Tasks
- update MCP tool naming and descriptions to advertise `zuraffa_make`
- ensure MCP command translation invokes `zfa make`
- update public examples, generated comments, sample files, and example project artifacts
- ensure docs explicitly say architecture skeleton is generated by Zuraffa, implementation follows afterward

### Exit criteria
- no active/public docs teach `zfa generate`
- no active/public AI surface prefers anything except the v5 pipeline

---

## M3. Legacy residue removal

### Goal
Remove confusing pre-v5 leftovers from active/public surfaces.

### Tasks
- remove obsolete docs that only describe removed commands or dead flag combinations
- remove placeholder/no-op public surfaces from the default product story
- update example artifacts and comments away from removed commands
- clean public HTML/static landing surfaces

### Notes
Historical migration docs may still mention removed commands explicitly, but only in migration context.

### Exit criteria
- repo grep over active/public surfaces finds no legacy residues except intentional migration/history files

---

## M4. `.zfa/` operationalization

### Goal
Make `.zfa/` a practical agent memory surface.

### Tasks
- define expected contents for `plans/`, `runs/`, `blueprints/`, `decisions/`, `manifests/`, and `context.json`
- add docs and examples for blueprint/decision usage
- ensure run artifacts and context are useful enough for a second agent session to continue work confidently

### Exit criteria
- `.zfa/` is documented and used as a first-class project memory contract

---

## M5. `zik_zak` proof migration

### Goal
Use `zik_zak` as the downstream proof that v5 works at app scale.

### Tasks
- migrate `zik_zak` config and guidance to v5
- seed `.zfa/` in `zik_zak`
- migrate old planning/spec references to the canonical pipeline
- prove one pilot feature slice with v5 regeneration
- validate adaptive-layout migration on a limited but real macOS/mobile slice

### Exit criteria
- `zik_zak` has a concrete migration branch plan and one proven pilot slice

---

# SHOULD

## S1. `zfa migrate`

### Goal
Automate the common migration path for real projects.

### Scope
- rewrite legacy `.zfa.json` to v5 shape
- detect/update old command references in project guidance
- seed `.zfa/`
- emit a migration report with manual follow-up items

---

## S2. `zfa doctor`

### Goal
Diagnose architecture and project-health issues.

### Scope
- stale generated files
- missing generated layers
- DI completeness
- `.zfa/` integrity
- root/config issues
- architecture boundary drift

---

## S3. `zfa diff`

### Goal
Show regeneration changes before they are applied.

### Scope
- preview file additions/modifications/removals
- machine-readable summary for agents
- human-readable grouped diff

---

## S4. GraphQL decision

### Goal
Resolve whether GraphQL is a real supported surface or a deferred one.

### Options
- finish and support it for real, or
- keep it internal/hidden until it is real

### Exit criteria
- no ambiguous half-public GraphQL story remains

---

## S5. Adaptive-layout downstream validation

### Goal
Move platform-aware layout support from “framework primitive” to “proven app capability.”

### Scope
- migrate selected `zik_zak` screens to `layouts/` and shell resolver usage
- compare manual macOS views with the new layout model
- refine generator output based on real friction

---

## S6. Unified root resolution

### Goal
Remove duplicate project-root resolution logic.

### Scope
- consolidate onto one root resolver for CLI/programmatic/MCP paths
- remove local ad-hoc directory walking in runtime-critical code

---

# NICE TO HAVE

## N1. Better docs and recipes
- adaptive-layout cookbook
- custom usecase cookbook
- migration recipes from legacy Zuraffa usage
- AI agent examples for common feature types

## N2. Generation ergonomics
- faster incremental regeneration
- better progress reporting
- richer plan explanation surfaces

## N3. Carefully scoped helpers
- routing helpers
- realtime watch/subscription scaffolds
- blueprint scaffolding helpers

---

# Suggested execution sequence

## Wave 1 — cleanup and contract enforcement
- M2 canonical pipeline everywhere
- M3 legacy residue removal
- tests guarding active/public surfaces

## Wave 2 — release and memory integrity
- M1 release integrity
- M4 `.zfa/` operationalization

## Wave 3 — downstream proof
- M5 `zik_zak` pilot migration

## Wave 4 — productivity features
- S1 `zfa migrate`
- S2 `zfa doctor`
- S3 `zfa diff`
- S4 GraphQL decision
- S5 adaptive-layout refinement
- S6 unified root resolution
